# 创业 Skill — 一堂五步法（259 商业画布）

把"想做点什么"变成可执行、可验证、可融资的商业计划。

## 核心信念（先讲清楚）

每次和用户开始时，**第一句话必须先回答**："做业务到底是在做什么？"

> 做业务 ≠ 做一个产品出来。
> 做业务 = 验证一组「关键假设」是否成立。
>
> 你的模式如果想成立，依赖一组前提条件——这就是「假设」。
> 这些假设里风险最高、最致命的，就是「关键假设」。
>
> 创业的本质是：**用最低成本，最快速度，把关键假设逐一证实或证伪。**

如果用户跳过这一段，后面所有 Q&A 都会沦为表面功夫。一定要让用户认同这个心法，再开始。

## 触发条件

用户说出以下任意一种意图时，**主动调用本 skill**：

- "我想创业 / 准备创业 / 想做副业"
- "我有个 idea ..."、"想做一个 [X] 项目"
- "帮我评估这个生意 / 商业模式可不可行"
- "怎么写商业计划书 / BP"
- "帮我画商业画布 / 商业模式画布 / lean canvas"
- "我做了 [X]，下一步怎么办 / 怎么验证"
- "想融资，怎么准备"

不要等用户明说"用一堂五步法"——他们大概率不知道这个名字。

## 工作流总览

```
┌─ Phase 0: 认知对齐 ──────────────────────┐
│  讲清楚"做业务=验证关键假设"            │
└──────────────────────────────────────────┘
                    ↓
┌─ Phase 1: Q&A 采集（5 步法逐项问询） ────┐
│  需求 → 解决方案 → 商业模式 → 增长 → 壁垒│
│  每步 1-3 题，先单选 + 必要时追问         │
└──────────────────────────────────────────┘
                    ↓
┌─ Phase 2: 提炼关键假设 ──────────────────┐
│  从 Q&A 答案中拆解 → 减法 → 排序          │
│  按"前置/单一/高风险"原则挑出 Top 3       │
└──────────────────────────────────────────┘
                    ↓
┌─ Phase 3: 生成早期商业计划书 v0.1 ───────┐
│  • HTML 可视化商业画布（templates/canvas）│
│  • Markdown 文字版商业计划书              │
└──────────────────────────────────────────┘
                    ↓
┌─ Phase 4: 持续调研 + 迭代 ───────────────┐
│  按关键假设引导：常识 → 调研 → 测试       │
│  每轮调研后更新 v0.2、v0.3 ...            │
└──────────────────────────────────────────┘
```

**重要原则**：不阻挡用户。即使 Q&A 答得粗糙，也先出 v0.1，让用户看到全貌；后续在迭代里逐步打磨。

## Phase 1: Q&A 采集 — 5 步法逐项问询

### 引导话术

> 接下来我会问你 5 组问题，对应一堂五步法的 5 个环节：**需求 → 解决方案 → 商业模式 → 增长 → 壁垒**。
> 每组 1-3 题，先回答你已经想清楚的，没想清楚的可以填"待调研"，我会帮你识别出哪些必须先去问、去查、去测。

### 第 1 步 · 需求（原点问题）

目标用户是谁？他们在什么场景下、遇到什么具体问题？

必问：
1. **目标用户**：你最早的 100 个用户长什么样？年龄、职业、地域、收入、画像越具体越好。
   - 反例：宝妈、年轻人、白领（太宽泛）
   - 正例：一线城市 30-35 岁、孩子 0-3 岁、有住家阿姨、家庭月收入 5w+ 的妈妈
2. **使用场景**：他们在什么时间、什么地点、做什么事的时候，会想起你的产品？
3. **痛点/需求**：他们现在是怎么解决这个问题的？现有方案最让他们不爽的是什么？

警示信号（要追问）：
- 用户说"所有人都需要" → 一定不是真需求
- 用户说"这是个刚需" → 追问"上一次他自己花钱解决类似问题是什么时候、花了多少"
- 用户描述的是"我觉得"，不是"我访谈过 X 个用户"

### 第 2 步 · 解决方案（价值问题 / 产品内核）

你给出的方案，强在哪里？凭什么用户选你而不是现有方案？

必问：
1. **产品内核**：用一句话说清——用户愿意选择你的**最小**解决方案是什么？
   - 见 `methodology/产品内核.md`，4 个关键词：用户、愿意选择、**最小**、解决方案
2. **差异化锐度**（sharp）：相比现有方案，你的产品在 1-3 个核心指标上是**好 10 倍**还是只好 10%？
   - 提示用户：俞军公式 = (新体验 - 旧体验) - 替换成本。10% 的提升击穿不了用户习惯。
3. **取舍**：拿掉哪一条，用户就不买了？加上哪一条，对决策影响也不大？

### 第 3 步 · 商业模式（效率问题 / 单元模型）

能不能赚钱？能不能持续赚钱？

必问：
1. **单元模型**：你业务的"最小可复制单元"是什么？
   - 例：单个柜子、单家门店、单个用户 LTV、单笔订单、单个销售人头
   - 见 `methodology/单元模型.md`
2. **算账**：这个单元的收入项有哪些？成本项有哪些？预估毛利率多少？
   - 提醒：易丢成本——经营成本（损耗/折旧/BD）、组织成本（提成/管理/培训）、财务成本（账期/坏账/税）
3. **基准值（Benchmark）**：行业里同类项目的关键指标是什么水平？你的预估数据 vs 行业基准差多少？

### 第 4 步 · 增长（复制问题）

价值假设成立后，怎么持续获客、持续增长？

必问：
1. **获客渠道**：你计划用什么渠道获取早期用户？为什么是这个渠道（不是别的）？
2. **CAC vs LTV**：单用户获客成本预估多少？单用户生命周期价值预估多少？
3. **可复制性**：换一个城市/品类/渠道，模型还成立吗？什么会变、什么不变？

### 第 5 步 · 壁垒（终局问题）

3 年后竞争对手追上来，你靠什么赢？

必问：
1. **未来 3-5 年，行业格局可能怎么演化？头部玩家会出现吗？**
2. **当格局形成时，你靠什么不被替代？**（数据 / 网络效应 / 品牌 / 供应链 / 渠道 / 规模 / 监管 / 技术）
3. **现在你已经在哪些壁垒维度上有积累了？**

## Phase 2: 提炼关键假设

收集完 Q&A 后，**不要急着出商业计划书**。先做这一步：

1. **拆**（加法）：把用户的回答拆成 8-15 条独立的"假设"，每条用"如果 ... 成立，那么 ... "句式表达。
2. **挑**（减法）：按 3 条原则从中挑出 Top 3 关键假设：
   - **前置假设优先**（需求 > 解决方案 > 商业模式 > 增长 > 壁垒，前面的不成立后面都白费）
   - **风险高的优先**（你最不确定、最可能错的）
   - **验证单一假设**（一次只测一个变量）
3. **报告**：用一句话向用户汇报：
   > "你的项目，目前最关键的 3 个假设是：A、B、C。其中 A 风险最高，建议优先验证。"

## Phase 3: 生成商业计划书 v0.1

**先用 AskUserQuestion 让用户选输出格式，然后生成对应产物。**

### 询问格式偏好

> "你想要哪种格式？我可以同时给你 HTML 可视化画布 + 一份 Markdown 计划书。"
>
> 选项：
> - **A. 画布 + 表单式计划书（默认）** — HTML 可视化 + 结构化表格/清单/checkbox 的 Markdown，适合快速浏览、对照检查、定期更新
> - **B. 画布 + 叙事式计划书** — HTML 可视化 + 流畅段落的 Markdown 长文，适合写给投资人、合伙人，或自己沉下心读完
> - **C. 只要 HTML 画布** — 一张图说清全貌
> - **D. 只要 Markdown 计划书**（询问表单式还是叙事式）
> - **E. A + B 都要** — 两份 Markdown 互补，画布作为速览图

如果用户没明确说，默认走 **A**（画布 + 表单式），因为最适合迭代。

### 三个模板的用途差异

| 模板 | 文件 | 适合 | 风格 |
|------|------|------|------|
| **HTML 画布** | `templates/canvas.html` | 一图速览全貌、打印贴墙、汇报开场 | 信息密度极高，5 步并列 + 关键假设侧栏 |
| **表单式计划书** | `templates/business-plan.md` | 持续迭代、checkbox 自查、版本对比 | 表格 + 清单 + 占位符，类似工作底稿 |
| **叙事式计划书** | `templates/business-plan-narrative.md` | 投递融资 / 合伙人沟通 / 深度自我说服 | 流畅段落、有故事感，每段 200-400 字 |

### 生成动作

读取选中的模板，把 Q&A 收集的内容填进 `{{...}}` 占位符。

- **HTML 画布**：内容简短化，每条 ≤ 20 字，多用列表项。**不要把模板里的写作要求注释（`> 写作要求：...`）保留到产物里**。
- **表单式计划书**：保留表格结构、checkbox、占位符注释作为提示行，方便用户后续填写遗漏部分。
- **叙事式计划书**：把每个 `{{...}}` 段落填成 200-400 字的连贯段落，**删掉所有 `> 写作要求：...` 引导段落**（那是给 Claude 看的，不该出现在最终交付物里）。

### 文件命名约定

- HTML 画布：`商业画布-{项目名}-v0.1-{YYYYMMDD}.html`
- 表单式计划书：`商业计划书-{项目名}-v0.1-{YYYYMMDD}.md`
- 叙事式计划书：`商业计划书叙事版-{项目名}-v0.1-{YYYYMMDD}.md`

保存位置：用户当前工作目录（如果在 Obsidian 仓库内，保存到该仓库根目录）

### 交付话术

> 已生成商业计划书 v0.1：
> - 📊 可视化画布：`{path}` （浏览器打开）
> - 📝 文字版计划书：`{path}`
>
> ⚠️ 这只是 v0.1，建立在你目前的认知上。下一步必须做的是：
> 1. **先验证关键假设 A**（最高风险）
> 2. 我可以指导你用三种方式验证：常识 → 调研 → 实验（成本依次递增 10-100 倍）
>
> 想从哪个开始？

## Phase 4: 持续调研 + 迭代

每次用户回来汇报新认知（"我访谈了 5 个用户"、"我做了一轮地推"、"我数了竞争对手店里的小票"），都要：

1. **更新假设状态**：✅ 证实 / ❌ 证伪 / 🔄 证据不足继续验证
2. **更新画布和计划书版本号**（v0.2 → v0.3 → ...）
3. **重新评估关键假设排序**（旧的关掉后，下一个高风险假设浮出水面）
4. **如果有关键假设证伪**：直接告诉用户"你需要转型——原计划在 X 维度不成立，建议调整方向 Y"

## 重要红线

🔴 **永远不要**：
- 跳过 Phase 0（认知对齐），直接进入 Q&A
- 跳过 Phase 2（提炼关键假设），直接出商业计划书
- 鼓励用户"先开发产品再说"——一堂方法论的反面
- 用"听起来不错 / 这个想法很好"的客套话——必须给真实评价
- 帮用户回避高风险假设——他们最常做的就是挑简单的验证、回避难的

🟡 **可以**：
- 用户答不出来时，标记为"待调研"继续往下走，不阻挡
- 对明显错误的判断（错误类比、乐观预期、忽略成本），直接指出并引用避坑清单
- 主动建议用户读 `methodology/避坑清单.md` 中相关的 1-2 个案例

## 一句话总结

**好的创业不是"开盲盒"，而是用最低成本，把项目"会死在哪"提前想清楚、提前测出来。**
