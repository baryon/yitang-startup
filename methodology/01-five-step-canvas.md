# 一堂五步法 + 关键假设三板斧

## 五步法（259 商业画布的 5）

每个业务持续发展中，最关键、最容易致命的 5 个难题：

| 步骤 | 别名 | 核心问题 |
|------|------|---------|
| 1. 需求 | 原点问题 | 目标用户是谁、在什么场景下、遇到什么问题？ |
| 2. 解决方案 | 价值问题 | 你的产品差异化在哪里？凭什么用户选你？ |
| 3. 商业模式 | 效率问题 | 能否赚钱？能否持续赚钱？ |
| 4. 增长 | 复制问题 | 如何持续获客、持续增长？ |
| 5. 壁垒 | 终局问题 | 面对竞争，靠什么不被替代？ |

**前 3 步 = 价值假设**（产品对用户有没有价值）
**后 2 步 = 增长假设**（能不能复制扩张）

> **核心原则**：不要过早复制"未经验证"的商业模式。
> 因为复制本身就是给错误加杠杆，把错误放大 10 倍、100 倍。

## 关键假设：业务的真正内核

**【定义】** 假设 = 你的模式如果想成立，所依赖的所有关键前提条件。
**【定义】** 关键假设 = 所有假设里，**风险最高、对业务最重要**的那几条。

> 不要看业务表面的现象（数据、产品形态、用户评价），而是围绕「关键假设」推进业务。

### 案例：京东无人餐厅（500 万学费）
- 表面：机器人做菜，京东大厂背书，媒体报道，客户慕名而来
- 关键假设：机器人不只是营销噱头，而是真的能提升经营效率（更好吃 or 替代人工）
- 实际：机器对食材形状要求严格，反而需要更多洗切人员；机器做的菜并不好吃
- 结果：关键假设证伪 → 一年后关停

### 案例：2005 年 WEB 版打车
- 关键假设之一：网页版打车体验远好于路边招手
- 时代约束：当时无智能手机、无 GPS 普及 → 假设不成立 → 注定做不成
- 教训：idea 不值钱，背后的关键假设值钱

## 关键假设三板斧

```
第一板斧：加法（拆）
  把业务拆成一组假设
  ↓
第二板斧：减法（挑）
  从中提炼最关键的 Top 3
  ↓
第三板斧：验证（答）
  用最低成本回答这些假设
```

### 第一板斧：加法 — 把业务拆成假设

用 259 中的任一画布拆解。最常用是**5 步法**，逐步问：

1. 需求假设：用户、场景、问题成立吗？
2. 解决方案假设：你的产品真的解决了吗？
3. 商业模式假设：单元经济模型能跑通吗？
4. 增长假设：获客渠道有效吗？
5. 壁垒假设：长期能守住吗？

每一步通常拆出 2-4 条具体假设，整个项目大约 8-15 条。

### 第二板斧：减法 — 提炼关键假设

3 条原则：

#### 原则 1：前置假设优先
```
需求 ← 解决方案 ← 商业模式 ← 增长 ← 壁垒
（前面的不成立，后面的全白搭）
```
反思题：如果当下验证失败，是回退一步？还是全盘推翻？
如果答案是"全盘推翻" → 你的顺序搞反了。

#### 原则 2：验证单一假设
- 一次只测一个变量，避免"3 个月开发一个大版本同时验证 3 件事"
- 反例：一批货还没卖过，为了集中采购打 9 折屯了 50 万的货

#### 原则 3：风险高的优先
- 大家都担心的、回避不掉的，就是当前的关键假设
- 反例：创始人喜欢挑简单的验证（如增长策略），刻意回避高风险的商业模式问题

### 关键假设的 7 大获取渠道（CheatSheet）

如果刚进入新赛道，可以从这 7 个渠道快速提炼：

1. **行业失败案例库** — 别人死在哪里？
2. **行业老兵 1v1 访谈** — 他们最怕什么？
3. **竞争对手深度调研** — 头部玩家在解决什么问题？
4. **专业投资人/FA** — 他们的 due diligence 清单
5. **真实用户访谈（5-15 人）** — 现有方案最让他们不爽的是什么
6. **自我事前验尸** — 这个项目最可能死在哪里？
7. **MVP 真实运营数据** — 不知道哪条假设最关键，就跑一个粗糙版本看哪里崩

### 第三板斧：验证 — 高效回答关键假设

成本递增，但准确度也递增：

| 方式 | 成本 | 适用 |
|------|------|------|
| **常识** | ¥0 | 行业经验值（Benchmark）已存在，问对人就有答案 |
| **调研** | ¥几千-几万 | 通过竞争对手数据、用户访谈、专家访谈拿到答案 |
| **测试** | ¥几万-几十万 | 设计精益实验，用 MVP 测试 |
| **全量开发** | ¥百万+ | 上面三种都拿不到答案时，最后选项 |

> **公式**：验证成本 = 假设数量 × 验证方式 × 资源投入
> 每跨过一个维度，成本就是一个量级的提升。

## 操盘段位：X 型 vs Y 型

- **X 型负责人**：把业务理解为"做一个产品出来"，组团队、研发、上线、看反馈
- **Y 型负责人**：把业务理解为"验证一组关键假设"，围绕假设来推进每一步

> 95% 的 CEO 是 X 型，但能跑出来的 95% 是 Y 型。

## 真实案例：东南亚电商平台（节省 90% 成本）

- 创始人原计划：开发简版淘宝，需 100 万
- 一堂引导拆解：识别出 2 条关键假设
  1. 选品要符合当地支付能力
  2. 必须有规模化获客渠道
- 关键问题：**只验证这 2 条，需要开发产品吗？**
- 实际方案：用客服人工模拟微信拼团，测试选品 + 支付能力
- 结果：验证成本砍掉 90%，几万块完成一轮业务认知

## 一句话心法

> **拆 >>> 不拆。拆得不够好，差别没那么大；拆与不拆，是两种完全不同的做事路径。**
